Developer Guide

Add or repair harnesses without breaking CI.

This guide is for contributors editing missions, harness wrappers, target metadata, and generated documentation. It focuses on the contracts the CI workflow depends on.

Harness Contract

What every active target needs

Runnable wrapper

Each target path must contain a zlaunch.sh that can run the default matrix headlessly and accept an optional time-warp argument unless it declares a CI performance profile.

Stable result file

The run must produce results.txt with one compact result line per expected case. CI derives the expected count from the wrapper and rejects incomplete files.

Isolated execution

Grouped harnesses should expose --jobs and --port_base, use scoped teardown, and avoid overlapping MOOSDB or pShare port ranges.

Add a Target

Checklist for a new manual dispatch harness

1. Start from the nearest exemplar

For app-level checks, start near harnesses/cmgr_harnesses/H01-cmgr_unit or harnesses/obmgr_harnesses/H01-obmgr_unit. For moving behavior checks, start near a current behavior family with the same vehicle count and port-isolation shape.

2. Keep the mission verdict-owned

Prefer pMissionEval and mission-visible flags for the real verdict. Use shell checks for result completeness, wall-clock bounds, warning scans, and other constraints that are awkward inside MOOS.

3. Register the target

Add the target to config/harness_targets.json with key, path, harness, artifact, and families. Add full_modes: ["correctness"] only for curated high-signal gates.

4. Update generated docs

Add a catalog entry in docs/tools/build_pages.py when the family should be public on the site, then regenerate pages with python3 docs/tools/build_pages.py.

Validation

Commands to run before pushing

Metadata and workflow checks

python3 -m unittest discover -s tests
python3 scripts/harness_targets.py validate
actionlint .github/workflows/build_extend.yml .github/workflows/pages.yml

Selector smoke checks

python3 scripts/harness_targets.py select --mode family_run --family waypoint_behavior
python3 scripts/harness_targets.py select --mode specific_harnesses --targets cutrange_h01,waypoint_h01

Harness runtime checks

cd harnesses/<family>_harnesses/<target>
./zlaunch.sh --just_make --jobs=4 --port_base=25000 10
./zlaunch.sh --jobs=4 --port_base=25000 10

Do not run two harness batches at the same time if their MOOSDB or pShare port blocks could overlap. For CI-style grouped validation, use explicit non-default --port_base values.

Source Map

Where each responsibility lives

Area Responsibility
missions/ Reusable MOOS stem missions, launch scripts, behavior templates, and mission-local evaluation setup.
harnesses/ Case matrices, overlays, harness wrappers, result aggregation, and per-harness README documentation.
config/harness_targets.json Manual dispatch source of truth for active CI targets, families, artifact names, and curated full-mode membership.
scripts/ CI result summarizers, target selectors, repeatability helpers, benchmark sweeps, and scoped teardown helpers.
docs/tools/build_pages.py Static site generator for the catalog, user guide, technical guide, developer guide, and generated harness pages.

Manual CI

After code lands

  1. Use specific_harnesses for the exact changed target when validating a narrow fix.
  2. Use family_run when a change affects a family-level mission stem or shared case pattern.
  3. Use batch_family_run when several behavior families should be checked against one shared build.
