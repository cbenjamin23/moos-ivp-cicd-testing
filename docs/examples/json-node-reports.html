The JSON reports that no consumer can read

JSON node reports are a real interoperability feature. Publishing them is only half the change: every shared node-report parser must recognize the new representation.

The proposed edit

Why it looks reasonable

JSON output is useful for dashboards and non-MOOS consumers. Adding a configuration option and converting the outgoing report looks self-contained because pNodeReporter publishes valid JSON and its legacy output path remains intact.

Explanation

What the failure tells us

What changed

After JSON publication was added, a receiving component could be handed a syntactically valid report that the shared parser interpreted as the legacy CSP format, yielding an invalid empty NodeRecord.

What caught it

The CTest sends a complete JSON vehicle report through the same generic parser used by consumers and checks the resulting name, position, motion, mode, and status fields.

Open the protecting test

Safer approach

The follow-up upstream change dispatches brace-wrapped input to string2NodeRecordJSON() and retains the CSP path for existing reports, making the new format additive rather than a compatibility break.
