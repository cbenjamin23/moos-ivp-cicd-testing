Example 03 · C++ unit test · debugging

The coordinate cleanup that swaps X and Y

A conventional output-argument order looks more natural, compiles cleanly, and returns plausible coordinates—but puts each value in the wrong field.

Verified failure Baseline 29ea136b6d Local mutation; no commit created

The proposed edit

Treat the geodesy outputs as X then Y

Why it looks reasonable

Most APIs return Cartesian coordinates in X/Y order. Reordering the two output arguments makes the call read the way a maintainer would expect, and the types provide no warning because both arguments are doubles.

Explanation

What the failure tells us

What changed

This MOOSGeodesy call uses the less obvious Y/X output order. Swapping the arguments still produces finite, believable numbers, but pNodeReporter serializes the wrong local position.

What caught it

The pNodeReporter CTest checks the numeric result in both conversion directions. Merely checking that X and Y exist would let this edit pass.

Open the protecting test

Safer approach

Keep the established Y/X argument order. If the call is wrapped for readability, give the wrapper named X and Y outputs and preserve a numeric round-trip test.
