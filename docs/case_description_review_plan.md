# Harness Case Description Review Plan

## Goal

Review the case descriptions for all 67 configured harness targets and 1,455
cases so each description states:

1. the unique input, configuration, event, or code path under test; and
2. the behavior or contract the case is intended to exercise; and
3. the concrete observation that actually determines whether the case passes.

The harness README is the source of truth. The website generator extracts case
descriptions from the README case bullets, so generated
`docs/harnesses/*.html` files should not be edited by hand.

## Writing Standard

Prefer one or two short sentences in this form:

```text
<Unique setup or stimulus>, exercising <specific behavior or contract>;
passes when <specific observable and value>.
```

Good harness evidence includes exact result fields, MOOS variables, posted
values, counters, state transitions, timing bounds, tolerances, process exits,
or expected absences. Use the evidence that actually controls the verdict.

Example:

```text
Places one obstacle in the center of the transit lane to exercise normal
avoidance. Passes when arrived=true, encounters=1, and collisions=0.
```

For expected-negative scenarios, distinguish the harness verdict from the
mission outcome:

```text
Disables obstacle-alert delivery to exercise the no-alert path. The harness
passes when the vehicle arrives after one encounter and one collision.
```

Avoid vague endings such as "works," "behaves correctly," "arrives cleanly,"
"produces a usable result," or "passes as expected" when the evaluator exposes
a more precise condition. Do not invent a criterion from a case name; confirm
it in the case patches, mission evaluator configuration, launcher checks, or a
representative result row.

Use **tests** or **verifies** only when the grading condition distinguishes the
intended behavior from a plausible regression. Use **exercises** when the case
enters a configuration or code path but its evaluator checks only a broader
outcome. Record that mismatch in the coverage-gap table rather than making the
description claim more than the case proves.

## Two-Pass Approach

### Pass 1: description and evidence audit

Review every case against its launcher mapping, patches, evaluator, and a
representative result. Rewrite the README description to state the exact setup,
intended behavior, and actual grading evidence. Add any insufficient or
non-discriminating evaluation to the coverage-gap table below.

### Pass 2: case strengthening

Use the coverage-gap table as the worklist for later test changes. Prefer
discriminating fixtures, exact payload or state comparisons, and mutation
probes that demonstrate the case fails when the intended behavior is removed.
After strengthening a case, rerun it and update its description from
**exercises** to **tests** or **verifies** when warranted.

## Completion Criteria

A harness is complete when:

- every case exposed by `zlaunch.sh` has one matching README description;
- each description identifies what is unique about the case;
- each description names the actual pass/fail evidence;
- expected-negative cases clearly explain why `grade=pass` is correct;
- repetitive setup details are omitted unless they distinguish the case;
- the generated harness page contains the revised wording; and
- repository invariants pass.

## Review Workflow

1. Open the harness `zlaunch.sh`, README, relevant case patches, evaluation
   configuration, and representative `results.txt` rows.
2. Confirm that the README case inventory matches the launcher case inventory.
3. Rewrite the README case bullets using the writing standard above.
4. Run focused cases when the evaluator evidence is unclear or the description
   would otherwise rely on an assumption.
5. Regenerate the website with `python3 docs/tools/build_pages.py`.
6. Verify the generated harness page, then run
   `python3 scripts/check_repo_invariants.py`.
7. Check the README and Page boxes below.

The README box means the full case inventory was reviewed against actual test
evidence. The Page box means the generated page was regenerated and checked.

## Pass 2 Attack Plan

The coverage-gap table is a triage backlog, not a requirement to preserve and
expand every existing case. Before changing a fixture or evaluator, assign the
gap one of these dispositions:

- **Strengthen harness** when live MOOS mail, process interaction, helm state,
  vehicle motion, or timing is essential to the contract.
- **Move or add CTest coverage** when the intended contract is a deterministic
  parser, calculation, state transition, or payload-construction rule that can
  be demonstrated directly without a mission.
- **Merge or remove** when another case already proves the same contract, the
  fixture does not reach a distinct code path, or the behavior is not an
  intended public contract.
- **Keep as an exercise** when the scenario remains useful integration
  coverage but stronger grading would be disproportionately brittle. Retain
  the honest wording and document why broader evidence is acceptable.
- **Defer** when the contract first needs an upstream design decision,
  instrumentation, or stable performance envelope.

Use the Status column to record `Triaged: strengthen`, `Triaged: CTest`,
`Triaged: merge/remove`, `Triaged: keep`, or `Deferred`, followed by `Resolved`
only after the chosen change and its validation are complete.

Track ordinary expected-negative work separately from behavioral gap closure.
Requiring the exact rejection state, error, or payload is **oracle hardening**:
it prevents an unrelated failure from passing but does not add a new behavior
path. Count a **behavioral gap** as resolved only when the case now observes
the distinguishing effect of its unique configuration or stimulus.

### Decision Rules

Prioritize a finding when a plausible implementation regression can currently
pass, especially for a core app contract. Prefer exact payloads, counts,
ordered state evidence, numeric tolerances, and explicit absence flags over
additional end-state assertions.

Prefer a domain event or state transition as the evaluator lead. Use ordered
event-driven phases when a contract has multiple stages. Use a fixed timing
checkpoint only when elapsed time is part of the contract or when the case must
prove that an event did not occur before a documented deadline.

Before retaining a case, answer all four questions:

1. What unique contract or failure mode would disappear if this case were
   removed?
2. Is a mission harness the smallest reliable layer that can prove it?
3. What deliberate mutation or broken fixture should make the improved case
   fail?
4. Can the evidence remain deterministic across supported systems and normal
   time-warps?

If question 1 has no specific answer, merge or remove the case. If question 2
is no, prefer CTest. If questions 3 or 4 have no practical answer, keep the
case only as an explicitly broad exercise or defer it.
