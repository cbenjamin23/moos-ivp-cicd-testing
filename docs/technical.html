Technical Overview

How the CI/CD pipeline is organized.

The repository treats MOOS-IvP stem missions as executable regression tests. It keeps reusable mission stems separate from case overlays, then runs those cases in CI with a shared launch and grading pattern.

Core Terms

Vocabulary for the rest of the site

Mission stem
The reusable base mission: MOOS communities, launch files, behavior templates, simulator setup, and map geometry shared by many checks.
Harness case
One scenario layered on top of a stem. A case changes only the inputs needed to test a specific behavior, edge condition, or regression risk.
Verdict
The pass or fail decision for a case. The mission usually reports the main verdict, and shell checks add file-completeness, timing, and warning-log constraints.
Matrix and artifact
A matrix is GitHub Actions running several selected harnesses as separate rows. An artifact is a file bundle shared or saved by the workflow, such as the built workspace or a harness results.txt.

Core Idea

A mission stem defines the common world: MOOS communities, simulator apps, behavior files, launch scripts, and baseline geometry. Harness cases apply small patches over that stem so one scenario can change without copying the whole mission.

Launch Model

Harness zlaunch.sh wrappers own case selection, patch application, result collection, and port isolation. They delegate the actual mission startup to shared xlaunch.sh, which keeps local and CI launch behavior aligned.

Verdict Model

pMissionEval owns the mission-visible verdict where possible. Shell-side checks add constraints that are easier to evaluate outside MOOS, such as wall-time ranges, result completeness, and warning-log scans.

Pipeline Layers

Why there are multiple harness types

Unit-style mission checks

Small stationary or controlled missions verify app outputs directly: detection, absence, report values, alert requests, filters, and lifecycle messages.

Moving integration checks

Short corridor missions verify that app outputs lead to correct downstream behavior: safe arrival, avoidance engagement, zero collisions, and expected fail modes.

Behavior correctness checks

Behavior-focused missions isolate behavior-owned contracts, such as alert requests, helper flags, lifecycle state, and resolution evidence.

Performance and endurance checks

Longer missions apply sustained pressure and add timing, warning-log, and repeatability constraints around mission-owned verdicts.

CI Execution

How GitHub Actions uses the repo today

Select

scripts/harness_targets.py converts manual workflow inputs into a runtime matrix. The source of truth is config/harness_targets.json, which maps target keys to harness paths and families.

Build Once

The workflow resolves the requested moos_ivp_ref to an exact SHA, restores or creates the build cache, builds MOOS-IvP and this repo, then uploads the built workspace as an artifact.

Fan Out

The runtime-harnesses matrix downloads that one build artifact. Each row runs one harness on its own GitHub runner, uploads results.txt, and fails if the expected case matrix is incomplete or mismatched.

Workflow Semantics

Selection, concurrency, and reuse

Selection is metadata-driven

Manual inputs are normalized into target keys before any runner starts. The selector owns validation, active-target filtering, and runtime-matrix shape; the User Guide owns the exact input strings.

Concurrency protects intent

The workflow concurrency group includes the workflow, branch, dispatch mode, and selected family/target list. A newer run with the same intent replaces the older one, while different family sets can run independently.

Build reuse has two layers

The cache avoids rebuilding the same MOOS-IvP SHA and repo source across workflow runs. The artifact shares one completed build across all runtime jobs inside a single workflow run.
