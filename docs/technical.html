Technical Overview

How the coverage workspace is organized.

The repository has two complementary coverage layers: CTest targets for source-level MOOS-IvP component behavior and mission harnesses for live MOOS app, behavior, and motion scenarios.

Core Terms

Vocabulary for the rest of the site

CTest target
A GoogleTest executable registered with CTest. Use it for component behavior that does not need a mission scenario.
CTest family
A workflow selector for one source-level component group, such as geometry, mbutil, or behaviors_marine.
Mission stem
The reusable base mission: MOOS communities, launch files, behavior templates, simulator setup, and map geometry shared by many checks.
Harness case
One scenario layered on top of a stem. A case changes only the inputs needed to test a specific behavior, edge condition, or regression risk.

CTest Coverage

tests/cpp/ holds GoogleTest executables for source-level logic: parsers, geometry, IvP functions, contact records, behavior helpers, app utility classes, and other component contracts.

Mission Harnesses

Harnesses run live MOOS missions when the behavior under test depends on process startup, MOOSDB traffic, vehicle motion, mission timing, or interactions between apps and behaviors.

Separate Selectors

CTest families select source-level component groups. Harness families and target keys select live mission workloads.

Coverage Layers

Why the workspace uses more than one test style

CTest component checks

Fast checks exercise library, app, behavior, and utility logic directly without launching a mission or allocating MOOS ports.

Unit-style mission checks

Small stationary or controlled missions verify app outputs directly: detection, absence, report values, alert requests, filters, and lifecycle messages.

Moving integration checks

Short corridor missions verify that app outputs lead to correct downstream behavior: safe arrival, avoidance engagement, zero collisions, and expected fail modes.

Behavior correctness checks

Behavior-focused missions isolate behavior-owned contracts, such as alert requests, helper flags, lifecycle state, and resolution evidence.

Performance and endurance checks

Longer missions apply sustained pressure and add timing, warning-log, and repeatability constraints around mission-owned verdicts.

Metadata

Selectors and invariant checks

CTest registry labels

tests/cpp/ target definitions use labels for CTest registry metadata and suite-level reporting. The workflow exposes only the top-level family selectors.

CTest invariants

scripts/check_cpp_tests.py checks that CTest registrations, labels, source references, discovery placeholders, and optional upstream bucket coverage stay consistent.

Harness targets

config/harness_targets.json maps target keys to harness paths and families so grouped mission validation can select the right harness set.

Source Orientation

Where to look in the repository

  1. tests/cpp/ contains the CTest coverage layer.
  2. scripts/check_cpp_tests.py contains the CTest invariant checker.
  3. missions/ contains reusable stem missions.
  4. harnesses/ contains case overlays, harness wrappers, and per-harness documentation.
  5. config/harness_targets.json contains mission harness target metadata.
  6. docs/tools/build_pages.py generates this website from repository metadata.
